优漾短剧 2.0：从剧本解析到连续图片与视频输出

YoYoung Shorts 优漾短剧不是单点模型入口，也不是把几个接口拼在一起的壳。它更像一套把剧本、角色、场景、道具、分镜脚本、连续图片和视频结果串成完整生产链路的 AI 短剧工作台。

由一名独立开发者持续构建。 当前仍处于首尾打磨阶段。出于个人精力和产品成熟度考虑，现阶段不会开放相关源码；后续会在产品更稳定后，考虑开放部分基础能力展示版，用于社区交流与展示。

为什么值得关注

现在 GitHub 上不缺“接模型”的工具，真正稀缺的是把创作流程组织起来的产品。

关注它，不是因为模型多，而是因为工作流是连着的

  • 剧本可以继续推进到资产、分镜、图片和视频，而不是在某一个按钮前停住
  • 角色、场景、道具会持续参与后续生成，而不是每一步重新描述
  • 图片一致性不是单张偶然结果，而是单图、多图叙事、分镜脚本共用的一条连续创作链
  • 结果会沉淀到历史记录里，便于回看、比较和继续扩展

它和常见同类工具的差别

  • 很多工具更像模型导航页，这里更像创作工作台
  • 很多工具要求先写复杂提示词，这里更强调先有想法，再逐步组织
  • 很多工具只停在图片层，这里会继续把脚本和图片推进到视频结果
  • 很多工具结果生成完就结束，这里更强调沉淀、复用和继续延展

功能区与核心能力

这里不是简单列按钮，而是解释为什么这些功能组合在一起，才构成产品真正的价值。

核心功能

1. 从剧本和项目开始

输入剧本、主题或者剧情片段后，系统会把它继续组织进后续创作流程，而不是停留在一次性提示词调用。

  • 降低短剧创作的提示词门槛
  • 更适合不会写复杂提示词的用户
  • 把灵感快速推进到后续脚本和画面阶段

2. 自动组织系列化分镜脚本

不是只写一段文案，而是会继续拆成不同镜头、不同景别、不同动作阶段的系列化脚本，方便后续连接图片和视频。

  • 自动组织镜头内容
  • 脚本不是终点，而是后续可执行骨架
  • 更适合短剧和连续内容创作

3. 图片一致性与资产锚定

角色、场景、道具不是一次性提示词，而是持续进入后续生成流程。单图、多图叙事、分镜脚本会共享同一条连续创作链。

  • 单角色一致性更容易被感知
  • 多图叙事更像连续画面，而不是独立结果
  • 资产中心是工作流锚点，不是摆设

4. 图像到视频闭环

分镜和图片不会停在静态图层，而是可以继续推进到视频阶段，让整套产品更像工作台，而不是几个分散功能页。

  • 脚本可继续推进为视频
  • 图片可继续承接到视频生成
  • 视频结果能进入历史沉淀与复用

Workspace：从项目入口开始

2.0 把项目、素材库、专业工作台和系统配置收束在同一套创作入口里，让短剧创作围绕一个项目持续推进。

Assets：角色、场景、道具作为锚点

资产矩阵不是装饰，而是工作流里的锚点系统。角色、场景、道具会在后续单图、多图、分镜脚本和视频阶段持续进入生成链。

Character：人物资产提示词

人物资产不只保存一张图，还会沉淀身份定位、服饰材质、面部识别点和连续性规则，让角色在后续镜头中更容易保持一致。

Scene：场景资产提示词

场景资产会把空间结构、入口方向、主体物、光线氛围和故事世界规则整理成可复用提示词，而不是只保留一张参考图。

Prop：道具资产提示词

道具资产会锁定形状、材质、尺寸比例、颜色分布和风格特征，降低后续分镜和视频生成中的关键物品漂移。

Story：多图叙事不是重复同一张图

多图叙事的重点不是多出几张图，而是把同一个剧情段落组织成连续画面，保留角色关系、场景氛围和动作推进感。

Library：资产库沉淀与复用

人物、场景、物品会作为后续镜头可复用的材料沉淀下来，帮助同一个故事世界继续扩展。

Script：自动组织系列化分镜脚本

Script 区不是只生成一段文案，而是会把内容组织成可继续执行的不同分镜脚本。每个镜头都可以继续连接图片和视频结果。

Video：图像与脚本继续推进到视频阶段

这里展示的是真正的视频功能区。它承接前面的图像和提示，让静态结果继续进入视频阶段，而不只是停留在图片层。

工作流程

YoYoung Shorts 优漾短剧最重要的不是某一个按钮多强，而是这些功能区之间是连着的。

剧本输入 → Parse 剧本解析 → Assets 资产矩阵 → Prompt 资产提示词 → Story 多图叙事 → Script 系列分镜 → Image 批量画面 → Video 视频结果

这条链路里，Assets 会持续进入各个阶段，角色、场景、道具不会在中途消失。也正因为如此，这套产品更像“创作工作台”，而不是几个散落的生成页面。

为什么不是单纯的图片展示页

如果这里只是堆很多截图，那么它看起来会像“效果图展板”。但对 YoYoung Shorts 优漾短剧来说，更重要的是把这些截图解释成一条连续关系：

  • 想法如何进入 Create
  • 资产如何进入后续 Story、Script 和 Video
  • 为什么多图叙事不是重复出图
  • 为什么脚本不是终点，而是继续推进视频的骨架
  • 为什么历史记录意味着结果可以沉淀和复用

也正因为如此，这个展示仓库最应该表达的不是“支持了多少模型”，而是“这套产品把创作过程组织成了什么样子”。

适用场景

短剧创作

更适合需要从一句剧情想法快速推进到分镜脚本、连续图片和视频结果的创作流程。

故事化内容可视化

适合需要把角色关系、场景氛围和连续画面组织出来的内容，而不是只生成单张图。

角色资产驱动创作

适合重视角色、场景、道具在后续阶段持续延续的内容生产方式。

从图像继续推进到视频

适合希望把静态画面进一步组织成视频结果，而不是在图片阶段停止的创作方式。

真正上传到 GitHub 之后，这些内容不会全部挤在一个 HTML 页面里，而是拆成：

  • README.md：首页和核心展示
  • FEATURES.md：详细功能说明
  • CASES.md：案例展示
  • ROADMAP.md：路线图

所以你现在看到的“文字说明不够多”，不是 GitHub 最终只能这样，而是我前一个版本的预览页还偏首页化。这一版已经往“更像别人完整仓库说明页”的方向补齐了。
